Este es uno de los proyectos más interesantes y formativos de la rama de Inteligencia Artificial en 42. Para resolver "Call Me Maybe", vas a construir lo que en la industria se conoce como un Structured Output Engine (un motor de salida estructurada).  PDFEl núcleo del desafío es que vas a trabajar con un modelo de lenguaje muy pequeño (Qwen 0.6B). Si solo le pides por favor en el prompt que te devuelva un JSON, fallará el 70% de las veces. Tu misión es hackear el proceso de selección de palabras (tokens) del modelo en tiempo real para que la probabilidad de que cometa un error de sintaxis sea exactamente cero.  PDFA continuación, tienes la guía de ruta detallada, paso a paso, con los conceptos teóricos que debes dominar y las fases de implementación.FASE 1: Fundamentos Teóricos (Lo que debes aprender primero)Antes de escribir una sola línea de código, necesitas entender cómo "piensa" un LLM por dentro. No necesitas saber la matemática profunda de los Transformers, pero sí el mecanismo de generación.1.1. ¿Cómo genera texto un LLM? (La tubería de tokens)Los LLMs no entienden palabras ni letras directamente; entienden números.  Tokenización: El texto de entrada se rompe en pedazos llamados tokens. Un token puede ser una palabra entera (" hola"), parte de una palabra (" mación") o un solo carácter ("{"). El SDK te da acceso a este mapeo a través de un archivo JSON de vocabulario.  PDFInput IDs: Cada token del vocabulario tiene un número entero asignado (su ID).  Logits (La clave de todo): Cuando le pasas los IDs al modelo, este no te devuelve una palabra. Te devuelve un vector gigante de números flotantes llamado Logits. Hay un logit por cada token existente en el vocabulario del modelo. Un logit más alto significa que el modelo "cree" que ese token es el que debería ir a continuación.  PDF1.2. Decodificación Restringida (Guided Decoding)Normalmente, los sistemas eligen el token con el logit más alto (codicia o Greedy Search). El problema es que el modelo Qwen 0.6B podría decidir que el siguiente token ideal es una palabra aleatoria en lugar de cerrar una comilla de un JSON.  PDFLa decodificación restringida consiste en interceptar esos Logits antes de que el modelo elija el siguiente token. Si tu regla dice que en este momento del JSON solo puede ir un número o un cierre de llave }, tú vas a buscar todos los tokens en el vocabulario que no sean números ni } y les vas a asignar un valor de menos infinito ($-\infty$) en el vector de logits. Al hacer esto, obligas matemáticamente al modelo a elegir únicamente entre las opciones válidas que tú has decidido. PDFFASE 3: El Núcleo del Proyecto – El Autómata Finito (FSM)Para saber qué tokens son válidos en cada paso de la generación, necesitas construir un Autómata Finito Determinarista (FSM) o una gramática que rastree el estado del JSON que estás construyendo. El modelo de salida que tienes que generar obligatoriamente debe lucir así:  PDFJSON{
  "prompt": "What is the sum of 2 and 3?",
  "fn_name": "fn_add_numbers",
  "args": {"a": 2.0, "b": 3.0}
}
Tu código de Python debe saber en todo momento en qué "estado" de la escritura se encuentra:  PDFEstado 0: Esperando abrir la estructura del JSON principal ({).  PDFEstado 1: Esperando escribir exactamente la cadena "prompt".  PDFEstado 2: Esperando los dos puntos : y luego el texto del prompt original.  PDFEstado 3: Esperando escribir exactamente la clave "fn_name".Estado 4: Esperando escribir el nombre de una de las funciones válidas que leíste en function_definitions.json (¡Aquí es donde guías al modelo a elegir una función real, no inventada!).Estado 5: Esperando escribir la clave "args".Estado 6: Abriendo el sub-JSON de argumentos y validando que los nombres de los parámetros y sus tipos correspondan exactamente a lo que esa función específica requiere.¿Cómo cruzar el estado con el Vocabulario del SDK?Carga el vocabulario del modelo usando get_path_to_vocabulary_json(). Este JSON es un diccionario gigante donde las claves son los tokens (cadenas de texto) y los valores son los IDs numéricos.En cada ciclo de generación de tokens, tu FSM evaluará: "Dado el estado actual, ¿cuáles son los caracteres textuales válidos a continuación?" Filtrarás el vocabulario entero y armarás una lista de IDs permitidos. Todos los que no estén en esa lista recibirán un valor de -inf en el tensor de logits.FASE 4: El Bucle de Generación AutorregresivoUna vez que tengas el filtro de logits basado en tu estado, implementarás el bucle de generación manual con el SDK:Tokenizar el Prompt Inicial: Convierte la consulta del usuario y el contexto base en input_ids iniciales usando encode().Bucle while o for hasta el token de parada (EOS):Pasa los input_ids acumulados a get_logits_from_input_ids(). Obtendrás los logits del siguiente token.Determina el estado actual de tu JSON generado hasta el momento.Aplica tu máscara de restricciones: pon en -inf los logits de los tokens prohibidos.Aplica un argmax (o selección del valor máximo utilizando numpy) sobre el vector modificado para obtener el ID del token ganador.Añade el ID seleccionado a tu lista de input_ids (para que el modelo lo tenga en cuenta en la siguiente iteración).Actualiza el estado de tu FSM analizando el nuevo token.FASE 5: Robustez y Requisitos de Evaluación (Asegurar el aprobado)Para defender con éxito el proyecto ante tus compañeros en la evaluación entre pares, debes asegurarte de cumplir con la sección IV.1 (General Rules) y las directrices adicionales:Tratamiento de Excepciones: Envuelve las lecturas de los archivos JSON de entrada (function_calling_tests.json y function_definitions.json) en bloques try-except.
